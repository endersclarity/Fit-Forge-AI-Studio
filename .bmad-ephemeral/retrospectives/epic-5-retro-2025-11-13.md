## Challenges and Learning Opportunities ðŸ§ 

### 1. **Tailwind v4 Compatibility Investigation (Story 5-1)**

Charlie (Senior Dev): "Story 5-1 hit an interesting bump with Tailwind v4.x, but the dev agent made the right call downgrading to v3.4.18."

**What Happened:**
- Initial implementation attempted Tailwind v4.x (new plugin structure)
- Required `@tailwindcss/postcss` instead of traditional plugin approach
- Assessment: v4.x would have introduced complexity for design system foundation

**Decision:** Downgrade to Tailwind v3.4.18
- **Rationale:** Stable, widely used, traditional PostCSS plugin structure
- **Benefit:** Compatibility with existing Vite ecosystem
- **Trade-off:** Deferred v4 upgrade to future optimization sprint

**Lesson Learned:** Sometimes "stable and proven" beats "latest and greatest" when laying foundations.

### 2. **WCAG Contrast Verification Implementation (Story 5-2)**

Alice (Product Owner): "The WCAG implementation in Story 5-2 had one medium-severity gap - no unit tests for the contrast calculation functions."

**What Happened:**
- Professional WCAG 2.1 implementation with gamma-corrected formulas
- All contrast ratios calculated and documented with recommendations
- **Gap:** No unit tests for `getContrastRatio()` and `meetsWCAG_AA()` functions

**Mitigation:**
- Risk documented: Accessibility calculations could regress without test coverage
- Recommended approach: Add tests in Epic 8 (Polish & Accessibility)
- Current status: Not blocking - implementation verified manually and documented

**Lesson Learned:** Even excellent implementations benefit from automated test coverage for critical utilities.

### 3. **Production Build Path Issues (Story 5-4)**

Dana (QA Engineer): "The font preload path issue wasn't caught during dev testing because the developer tested in development environment, not production build."

**What Happened:**
- First code review identified broken font preload paths in index.html
- Hard-coded paths `/node_modules/@fontsource/...` didn't match Vite's hashed output
- Problem: Static paths in HTML couldn't adapt to Vite's asset hashing

**Root Cause Analysis:**
- Development environment loads fonts directly from node_modules
- Production environment uses hashed filenames (e.g., `cinzel-latin-700-normal-Dkw14w9r.woff2`)
- Testing only in dev environment missed this mismatch

**Resolution:**
- Removed broken preload tags (simplest correct solution)
- Fonts load via CSS @import with font-display: swap
- No performance regression - CSS bundling handles optimization

**Lesson Learned:** Always test in production build mode before final submission. Development and production environments can have significant differences.

### 4. **Primitive Components Test Coverage (Story 5-3)**

Charlie (Senior Dev): "Story 5-3 created 123 tests, which is fantastic, but we should document which test patterns worked best for component testing."

**What Happened:**
- Comprehensive test suite for Button, Card, Input, Sheet components
- jest-axe accessibility audits integrated
- All variants tested (primary/secondary/ghost for buttons, glass/elevated for cards, etc.)

**Opportunity:**
- Document best practices for component testing patterns
- Create testing guidelines for Epic 6-8 components
- Reusable test templates for new primitive components

**Action for Future Epics:** Create shared testing utilities and patterns document.

### 5. **Documentation Maintenance Challenge (Story 5-2)**

Elena (Junior Dev): "The token documentation in Story 5-2 is amazing, but I'm wondering how we keep it in sync as design changes happen."

**What Happened:**
- 313-line comprehensive README documenting all tokens
- Design system specifications in tailwind.config.js, tokens files, and documentation
- Multiple sources of truth risk

**Mitigation Strategy:**
- Token files (colors.ts, typography.ts, etc.) are the source of truth
- Documentation is generated/maintained based on actual token files
- Tailwind config mirrors token definitions
- Keep documentation references pointing to actual file locations

**Lesson Learned:** Single source of truth architecture prevents documentation drift.
